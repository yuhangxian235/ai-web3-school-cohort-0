## 📝 三个关键理解

### ① 上下文窗口是什么
> 就像你和朋友的**关系记忆**——你们怎么认识、聊过什么、当前在聊什么，这些对话历史构成了你和 TA 之间的"上下文"。LLM 也一样，每次调用时能看到的所有消息就是它的上下文。

### ② Agent 跑 50 步为什么忘事
> Agent 每步都会往上下文里追加内容：用户输入 → LLM 思考 → 工具调用 → 工具结果。像聊天记录越堆越长。当总 token 超过模型的**上下文窗口上限**（DeepSeek 64K token），最早的内容被"截断"——Agent 就忘了最初的指令是什么，和你翻到聊天记录最前面发现已经看不到最初的对话一样。

### ③ 上下文满了怎么办
> **压缩上下文**，提取关键信息扔掉废话。具体方法：摘要压缩（把长对话总结成一段话）、滑动窗口（只保留最近 N 轮）、RAG（把旧信息存到外部知识库，用时检索）。对应记忆系统"记住关键的事，忘掉不重要的事"。

---

## ✍️ 一句话总结

LLM 的上下文就是短期记忆，有容量上限。Agent 不管理上下文就会"忘事"——这和真实对话一样，长对话需要总结和提取关键信息。

## 📝 三个关键理解

### ① RAG 三步走

| 步 | 名称 | 做什么 | 比方 |
|---|---|---|---|
| 1 | Embedding | 文档 → 数学向量 | 把书页拍成微缩胶片 |
| 2 | Retrieve | 问题转向量 → 找最相似文档 | 在胶片库里搜匹配页 |
| 3 | Generate | 搜到的文档 + 问题 → LLM | 翻到那页，照着回答 |

### ② 什么时候用 / 不用 RAG

| 用 RAG | 不用 RAG |
|---|---|
| 问私有数据（"我的合约 pay 函数干嘛的"） | LLM 本来就懂的（"Solidity struct 怎么定义"） |
| 知识会变化（"最新的 EIP 标准是什么"） | 通用常识（"比特币是什么"） |
| 需要精确引用的（"文档里第 3 章说了什么"） | 创意性回答（"帮我想一个合约名字"） |

> 💡 **核心判断：LLM 是闭卷考试，RAG 是开卷考试。** 翻书才找得到的答案 → 用 RAG。

### ③ StablePay × RAG 怎么用

把 PaymentProcessor 合约的函数签名、参数、返回值、NatSpec 注释存入向量库。Agent 问 "pay 函数干嘛的" → 检索出相关内容 → 给出准确回答，而不是瞎编。

---

## ✍️ 一句话总结

RAG = 检索 + 生成。先把文档转成向量存好，用户提问时搜出最相关的片段，拼进 prompt 让 LLM 回答。解决了"LLM 不知道你仓库里有什么"的问题。

## 📝 三个关键理解

### ① Agent 三要素

| 要素 | 是什么 | 比方 |
|---|---|---|
| **LLM**（大脑） | 做推理和决策 | 人的大脑 |
| **Tool Use**（工具） | 调用外部能力（查天气、查余额、发交易） | 人的手和工具 |
| **Loop**（决策循环） | Plan→Act→Observe→Replan，循环到任务完成 | 人做事的方式：想一下→做一下→看结果→调整 |

### ② 聊天机器人 vs Agent

> 聊天机器人：你问一句，它答一句。一轮结束。
> Agent：你给一个目标，它自己一步步做完——查数据、调工具、确认、执行——中间不需要你再说话。

### ③ StablePay 被 Agent 调用：9 步转账工作流

```
用户: "帮 Bob 转 50 USDC"
  ↓
1. 解析意图      → Bob 是谁？50 USDC 哪个链？
2. 查 Bob 地址   → 通讯录 / ENS / 历史
3. 查余额        → USDC 够不够？
4. 模拟交易      → estimateGas，算费用
5. 展示确认      → "转 50 USDC 给 Bob，Gas $0.3，确认？"
6. 用户签名      → 等待确认 ← human-in-the-loop
7. 广播交易      → 上链
8. 等待确认      → 监控 tx hash
9. 通知完成      → "已到账 ✅"
```

> 💡 这就是 human-in-the-loop 模式：Agent 干所有活，但关键步骤（签名/转账）必须人确认。

---

## ✍️ 一句话总结

Agent = LLM + 工具 + 循环。不是一问一答，而是拿到目标后自己规划步骤、调用工具、观察结果、调整计划，直到完成。StablePay 场景下，Agent 负责查余额、构造交易、监控上链——但签名必须用户拍板。

## 📝 三个关键理解

### ① MCP 三个角色

| 角色 | 是谁 | 做什么 |
|---|---|---|
| **Client** | Agent 这边 | 发起请求："查余额" |
| **Server** | 工具那边 | 暴露能力："我会查余额" |
| **Transport** | 中间层 | stdio/HTTP 传输数据 |

### ② 有 MCP vs 没 MCP

> 没 MCP：Agent 接 10 个工具 = 写 10 套适配代码
> 有 MCP：Agent 接 10 个工具 = 同一套协议，工具自己注册能力

就像 USB：没 USB 每个设备要自己的接口，有 USB 一个口插所有。

### ③ StablePay MCP Tools

```
Tool 1: get_balance(address) → 查 USDC 余额
Tool 2: get_payment_history(address, limit) → 查支付记录
Tool 3: create_payment_intent(amount, to) → 创建支付意图（不广播，等签名）
```

---

## ✍️ 一句话总结

MCP 统一了 Agent 和工具的通信标准。StablePay 通过 MCP 暴露合约能力后，任何 MCP 兼容的 Agent 都能直接调用——不用为每个 Agent 重写接口。
